Overview: top
  1. What is the Eclipse Project?

    The Eclipse Project is an open source software development project dedicated to providing a robust, full-featured, commercial-quality, industry platform for the development of highly integrated tools. It is composed of three subprojects, Platform, JDT - Java development tools, and PDE - Plug-in development environment. The success of the Eclipse Platform depends on how well it enables a wide range of tool builders to build best of breed integrated tools. But the real vision of eclipse as an industry platform is only realized if these tools from different tool builders can be combined together by users to suit their unique requirements, in ways that the tool builders never even imagined. The mission of the Eclipse Project is to adapt and evolve the Eclipse Platform and associated tools to meet the needs of the tool building community and its users, so that the vision of eclipse as an industry platform is realized.

  2. What is the Eclipse Platform?

    The Eclipse Platform is an open extensible IDE for anything and yet nothing in particular. The Eclipse Platform provides building blocks and a foundation for constructing and running integrated software-development tools. The Eclipse Platform allows tool builders to independently develop tools that integrate with other people's tools so seamlessly you can't tell where one tool ends and another starts.

  3. What is the Eclipse SDK?

    The Eclipse SDK (software developer kit) is the consolidation of the components produced by the three Eclipse Project subprojects (Platform, JDT - Java development tools, and PDE - Plug-in development environment) into a single download.

    Together these pieces provide a feature-rich development environment that allows the developer to efficiently create tools that integrate seamlessly into the Eclipse Platform.

  4. How is the Eclipse SDK licensed?

    The Eclipse SDK consists of software produced by the Eclipse Project combined with third party software from other open source projects. The software produced by the Eclipse Project is made available under the Common Public License. Third party components are made available under their own licenses. The detailed licensing information can be found in the Eclipse.org Software User Agreement (notice.html) file in root directory of the SDK download. Licensing for third party components is described in about.html files in the respective subdirectories.

  5. How is the Eclipse Project organized?

    The Eclipse Project is one of the top-level projects hosted at http://www.eclipse.org/. The Eclipse Project is organized into subprojects. Each of these in turn has one or more components.

    • For more information on the organization of the Eclipse Project, read the Eclipse Project Charter.
    • To learn more about the subprojects of the Eclipse Project, visit the Eclipse Project page and click on any of its subprojects.
    • For information on a specific component, see the subproject page that contains the component, and the component page.

  6. What is eclipse.org?

    Eclipse.org is a consortium of software-development-tool vendors formed to foster the development . The eclipse community was formed in order to create better development environments and product integration. The community shares an interest in creating products that are easily inter-operable, because they are based upon plug-in technology and a common platform. By collaborating and sharing core integration technology, vendors who build on the Eclipse Platform can concentrate on their areas of expertise and the creation of new technology.

  7. How do I find out about future releases of the Eclipse SDK?

    The Eclipse Project page contains a link to the plan outlining the goals for the next major release of the Eclipse Project.

    In addition, each component page provides information about work in progress.

Getting Started: top
  1. What build should I download?

    You should download either the latest release build or the most recent stable development build. These builds are available from the downloads page. The latest release will have fewer bugs but the most recent stable build will have all the latest features. If you are just getting started it is safer to work with the latest release so you won't be confused by random bugs in the latest stable development build. You should use only release builds for any kind of production work. A notice will be posted to the newsgroup when a new stable build is available.

  2. What does a green check mark or a red 'X' mean on the build download page?

    The green check mark and the red 'X' are rough estimates of the quality of the build.

    A green check mark means the build passed the following tests or is considered stable enough for use by the eclipse development team. A build will a green check can still have bugs and regressions!

    1. The plug-ings compile.
    2. The automated tests all pass.
    3. The build passes a five to ten minute human-driven sniff test.

    If the build fails any of the tests it is marked with a red 'X'.

  3. Which operating systems does the Eclipse Platform support?

    The 1.0 Release build of the Eclipse Platform was designed to run on Windows XP, Windows 2000, Windows 98, Windows ME and Red Hat Linux Version 7.1 (x86/Motif).

    The most recent 2.0 builds of the Eclipse Platform are designed to run on Windows XP, Windows 2000, Windows 98, Windows ME, Red Hat Linux Version 7.1 (x86/Motif and x86/GTK)), SuSE Linux 7.1 (x86/Motif and x86/GTK) and Solaris 8 (SPARC/Motif).

    More platforms are being added. For the information on which platforms are supported in a given build visit the downloads page. Each build indicates the platforms it runs on. For more information see the Eclipse Project plan located on the Eclipse Project page.

  4. I have downloaded the Eclipse SDK, but it won't start. How do I start eclipse?

    The number one reason is that the JRE cannot be located to run eclipse. You must have a Java Runtime Environment (JRE) installed on your computer. Eclipse requires version 1.3 or 1.4 of a Java 2 Standard Edition JRE. The Eclipse SDK does not ship with a JRE.

    We recommend that you download and install a Java developer kit. The developer kit will include a Java runtime that you can use to launch eclipse. Each build page contains a link to a page of suggested developer kit and Java runtime downloads that should work with the eclipse build. You can also find these links to commercial vendors that supply Java developer kits and Java runtime environments in the table below.     Once you have installed a Java developer kit, you can use the -vm command line argument to launch eclipse. For example, if you have installed a Java developer kit in c:\jdk1.3, you can launch eclipse with the following command line:

    eclipse.exe -vm c:\jdk1.3\bin\javaw.exe

    NOTE: Some Java runtimes and developer kits install themselves on the system path automatically. If this happens, you do not need to specify a JRE when you launch eclipse. If you want to use a different Java runtime or want to be sure of which Java runtime you are running, simply use the -vm argument and eclipse will use the Java runtime you specify.

  5. How do I write a simple "Hello World" program?

    To write a "Hello World" program follow these steps:

    1. Start eclipse.
    2. Create a new Java Project:
      1. File->New->Project.
      2. Select "Java" in the category list.
      3. Select "Java Project" in the project list. Click "Next".
      4. Enter a project name into the Project name field, for example, "Hello World Project".
      5. Click "Finish", the Java perspective will open.
    3. Create a new Java class:
      1. Click the "Create a Java Class" button in the toolbar.
      2. Enter "HelloWorld" into the Name field.
      3. Click the checkbox indicating that you would like eclipse to create a "public static void main(String[] args)" method.
      4. Click "Finish".
    4. A Java editor for HelloWorld.java will open. In the main method enter the following line.
      System.out.println("Hello World");
    5. Save using Ctrl-s. This automatically compiles HelloWorld.java.
    6. Click the "Run" button in the toolbar.
    7. You will be prompted to create a Launch configuration, select "Java Application" and click "New".
    8. Click "Run", the Hello World program will run. The console will open results, "Hello World".

  6. How do I compile my Java program?

    Eclipse has an incremental compiler which compiles every time you save a Java file. You will see this feature referred to as "autobuild".

    See Window->Preferences->Workbench->Perform build automatically on resource modification.

    With autobuild turned off, you can press the "Build" button (that appears on the toolbar) to compile your Java files. The "Build" action will appear on various other menus as well (for example, on the popups in the Navigator and Packages views).

  7. How do I run/debug my Java program?

    Select your Java program's main class in the Packages view and click either the "Run" or the "Debug" button. These buttons are located in the toolbar. Once you have launched your program you can easily launch it again by using the F9 key.

    When you run your program, it will execute at full speed. When you debug your program you will be able to control it using the debugger. This means you will be able to pause Threads, use breakpoints and investigate your program's internals. It also means your program will run slower due to the communication that must occur between the VM and the debugger.

  8. How do I use eclipse to access a CVS repository?

    To access a CVS repository you must have a repository location. To create a new repository location follow these steps:

    1. Open the CVS Repositories view (Window->Show View->Other...->CVS->CVS Repositories).
    2. Right click in the CVS Repositories view and select New->CVS Repository Location...
    3. Fill in the form detailing your CVS server's information. For example, we will use the information to connect to the Eclipse Project's public CVS server.
        Host: dev.eclipse.org
        Repository path: /home/eclipse
        User: anonymous
        Password: <leave empty>
        Connection type: pserver
    4. Click "Finish".

    Once you have a repository location, you will want to share your projects by releasing them to the repository. To share a project with a repository select Project->right click->Team->Share Project... You will be prompted to select the repository with which you would like to share your project. Once you have selected the repository click Finish. The Synchronize view will appear. When releasing new projects, you can simply right click on the project and select Commit. You will be prompted to enter a description. Once you have entered a description of your release, your code will be released to the CVS server.

    After you have shared your projects with a repository you can use the "Synchronize with Repository..." function to release code and to catch up with changes that others have made to the code in the repository.

    See Project->right click->Team->Synchronize with Repository...

    TIP: If commits to the repository fail, check your workspace/.metadata/.log file. If the following error is there, you should set the permissions on your CVS server directory to +x for everyone (this is a workaround).

  9. Where are the eclipse manuals?

    There are two ways to read the manuals.

    1. Read the manuals within the Eclipse SDK.
        To view these books open Help Contents from the Help menu and then select a book from the combo box.
    2. Read the manuals online. They are available in either HTML or PDF format.

  10. How do I ask questions?

    Questions that are not answered in the FAQ or in the manuals should be posted to the newsgroups. You will need a password.

    Please stick to technical issues on the newsgroups. Keep in mind that these newsgroups are public, so do not include any confidental information in your questions! You should also read "How to ask questions the smart way" by Eric Raymond before participating in the newsgroups.

    NOTE: Please submit bugs to bugzilla, not to the newsgroups. See the How do I report a bug? section of this document.

  11. How do I report a bug?

    The Eclipse Project uses bugzilla as its bug tracking system. Entering a bug report is as simple as filling in a web form on the eclipse bugzilla page. The first time you enter a bug you will need to create a new bugzilla account for yourself by providing an email address and choosing a password.

    But wait! Before you enter a bug report, search bugzilla to see if someone else has already entered a bug report similar to yours. If you find a bug report that outlines the problem you are seeing, you can simply annotate it with your comments to let the developers know that you have also hit the bug.

    Once you have searched bugzilla and not found anything, you can go ahead and enter a new bug report. Please read the bug writing guidelines located on the eclipse bug reporting page.

    To make your bug report more helpful include the following in all bug reports you submit:

    1. The Eclipse SDK build you are using, for example, Eclipse SDK 20011206
    2. Your computer's specifications (OS, memory, other pertinent info)
    3. The contents of your .log file (or lack thereof). This is especially important if you get a dialog that reports an internal error. See Where is the .log file?for additional instructions on finding your .log file.
    4. The Java runtime or development kit you are using to run eclipse (use java -version or java -fullversion)
    5. A description of what you were doing when you encountered the bug
    6. An explanation of how the observed behavior differs from the expected behavior
    7. Step-by-step instructions that repeatedly expose the bug, so that the developer can see it for themselves.

  12. Where is the .log file?

    The .log file is located in the workspace/.metadata directory.

    The .log file is used by the Eclipse Platform to log runtime errors. It is useful to include it in bug reports because it contains stacktraces that occur in plug-ins. When you report a bug, make sure to include your .log file!

  13. How do I make feature requests?

    Feature requests can be made through bugzilla.

    But wait! Just like with bug reports, you should first search bugzilla to make sure that nobody has already made a similar request. If the feature request already exists simply annotate it with your comments.

    When entering feature requests, please attempt to be very specific about the feature you would like, its use cases, and possible pitfalls that the implementor should avoid.

  14. How do I bring my Java files into eclipse?

    The best way to do bring your pre-existing Java files into eclipse is to create a Java project and copy your files into the project. You can copy the files into the project using a number of methods. You can:

    • Use the import facility, File->Import...
    • Copy the files into the project folder in the OS. Then right click on the project and select Refresh From Local.
    • (Win32 only) Drag the files from the desktop, a folder, or an Explorer window into the project.

    If you do not want to copy the files, you can create a Java project that is located at the folder that contains your Java source. For example, let's say you have 5000 source files in directory tree under c:\work. You can create a Java project and set its location to c:\work. To do this, uncheck the "Use default location" check box in the new project wizard to specify the location.

    NOTE: Eclipse does not allow project locations to overlap.
  15. When I delete a project eclipse asks if I want to delete the content. What does this mean?

    The Eclipse Platform stores your project contents in a directory. By default this directory is in the workspace directory and is named with the project name. When you delete a project, you are asked if you want to delete the contents of this directory.

    In many cases it is desirable to delete the contents and the project entirely. But when you have located your project in a location other than in the default location, you may want to keep your files, and simply remove the project from your Eclipse workspace.

    TIP: Be careful! If you have created a project in a location other than the default location you can delete files that you did not intend to.

  16. Eclipse shows my packages prefixed with a folder name (for example, sources.mypackage.MyClass). How do I fix this?

    You can tell eclipse which folders in a Java project are folders containing source by editing your Java project classpath:

    1. Open the Java Build Path properties page, Java Project->right click->Properties->Java Build Path.
    2. Select the Source tab.
    3. Select "Use source folders contained in the project".
    4. Click "Add Existing Folders..."
    5. Select your source folder and click "OK".
    6. Eclipse will ask if you want to change your output directory to /<project name>/bin, answer "Yes".
    7. Click OK in the Properties dialog.

  17. Why are my resources being copied to my project's output folder?

    When a project's build path is configured with an output folder, JDT generates class files into this folder and copies resources inside the packages to the output folder. This ensures that the resources will be found at runtime by the running Java application. To avoid the resource copying, you can do one of the following.

    • Do not use source folders and set the output folder to the project itself
    • Put your resources that should not be copied into a folder (for example, doc) that you do not add as a source folder to the Java build path of your project.

  18. Can I have an external source folder in my project?

    This is currently not supported, but each project can have a different location in the file system. If your source is split across multiple folders on the file system, you can define projects, each with a location corresponding to the appropriate folder in the file system.

  19. How do I use a different Java runtime (JRE) to run my Java program?

    1. Go to Window>Preferences>Java>Installed JREs.
    2. Click "Add".
    3. Enter a name for your new JRE. For example, JDK 1.3.
    4. Click "Browse" to locate the JRE home directory of the JRE you want to add.
    5. Once you have selected a JRE home directory, the JRE_SRC variable will default to the src.jar for the JRE you have selected. Some JREs are shipped with the source located in a file other than src.jar. If you need to set the JRE_SRC variable for your JRE, unselect the "Use default library" and browse your filesystem for the source file.

      WARNING: Some JREs do not ship with source. Setting the source file for JREs that do not ship with source can can cause the debugger to highlight incorrect source lines.

    6. Click "OK" when your settings are completed.
    7. In the JRE list, select the JRE that you want to be the default JRE for all your projects.
    8. Click "OK" to finalize your changes.

    NOTE: It is possible to run with a different JRE than the default JRE. To check which JRE you are running with see Run->Run..., select your Launch configuration and click on the JRE tab.

  20. How do I use the "HotSwap" feature?

    In order to use HotSwap, you must debug your program with a Java runtime that supports HotSwap (some 1.4 level Java runtimes support this feature). If you do not already have a Java runtime that supports HotSwap, you should download and install one on your computer.

    1. Switch your runtime JRE to a HotSwap-enabled JRE by editing your launch configuration.
    2. Put a breakpoint in your code by double clicking in the sidebar of the Java Editor next to a line of Java code.
    3. Debug your program.
    4. When your breakpoint is hit, alter your code.
    5. Save using Ctrl-s.
    6. If autobuild is turned off, you will have to perform a build manually by clicking the "build" button that is located in the toolbar.
    7. After the build, the HotSwap occurs. The VM will be running your new code.

  21. Ant complains about the classic compiler. How do I fix this?

    Ant may give the following error message:
    "Cannot use classic compiler, as it is not available. A common solution is to set the environment variable JAVA_HOME to your JDK directory."

    To workaround this problem, you can add tools.jar to the Ant classpath or change the compiler Ant uses to build Java files to be the compiler that ships with the Eclipse SDK.

    If you want to add tools.jar, go to Window->Preferences->Ant->Customize and add tools.jar. This option is not available in Eclipse version 1.0 and early builds of Eclipse 2.0.

    To change the compiler Ant uses, you have two choices:

    1. Add the following line to your script:
      <property name="build.compiler" value="org.eclipse.jdt.core.JDTCompilerAdapter"/>
      The problem with this solution is that the script gets hard-coded and it is likely that you will not be able to run the script outside of eclipse.
    2. Set this property on the wizard that runs the Ant script as follows:
      1. Select your script->right click->Run Ant...
      2. In the arguments field, add the following:
        -Dbuild.compiler=org.eclipse.jdt.core.JDTCompilerAdapter

    NOTE: These instructions are for the latest 2.0 builds. If you are running Eclipse version 1.0 or a 2.0 build from before June 21, 2002 you will need to replace "org.eclipse.jdt.core.JDTCompilerAdapter" with "org.eclipse.pde.internal.core.JDTCompilerAdapter".

  22. Does the Eclipse SDK ship with a GUI designer?

    No, the Eclipse SDK does not ship with a GUI designer.

  23. How do I use eclipse to debug a Java program running on another machine?

    To debug a remote Java program follow these steps:

    1. Start eclipse.
    2. Switch to the Java Perspective, Perspective->Open->Java.
    3. Create a new Java Project by clicking on the "Create a new Java Project" button in the toolbar.
    4. If you have source for your program, you can import it into the new project, so that the debugger shows source while you are debugging. This also allows you to set breakpoints in the remote program.
    5. Launch your Java program the remote machine. Make sure you add the following to the command line arguments so that the Java program will be debug enabled:
        -Xdebug -Xnoagent -Xrunjdwp:transport=dt_socket,server=n,suspend=n,address=8000 -Djava.compiler=NONE

      TIP: You can change the address value to specify any port you would like to use, just remember what you changed it to, you will need it later.

    6. Select the Java Project you have just created.
    7. Select Run->Debug...
    8. Select Remote Java Application and click "New"
    9. Enter the host and the port number of your remote Java program. Enter this information and click "Finish".
    10. The debugger will connect to the remote Java program and switch to the Debug perspective.

    TIP: This setup forces your Java program to wait until the debugger connects to it before executing. If you are debugging a program that does not need to be debugged during startup, you can change the command line argument "server=n" to "server=y". This is useful for debugging web servers and similar programs.

    TIP: You can use this technique to connect to Java programs you have launched on your local machine using the command line, simply leave the host as localhost in the Launch Configuration Wizard.

  24. I get an error dialog launching the debugger. What can I do?

    This answer to this question depends on the error message you get:

    • If the dialog says "Problems encountered launching the Java VM in debug mode", look at the reason for the failure to determine a possible solution. The reason is an error message produced by the VM. For example, the error message might say something like 
      	"Error occurred during initialization of VM
	Could not find -Xrun library: jdwp.dll"
      This error message indicates that the JRE you have specified to use for debugging does not support debugging, or cannot locate the proper DLLs required for debugging. Use a JRE that supports debugging or change the JRE setup to point to the correct JRE root where the jdwp.dll can be found. (Note: VMs before 1.3.x do not generally support debugging unless you have installed extra debugging DLLs).
    • If the dialog says "Timeout while waiting for VM to connect. Retry?", This means the debugger has not yet received a connection from the VM it has launched, and the VM has not reported any launching errors. In this case, retry a few times. It is possible that the VM needs more time to load/initialize itself.

  25. Why can't I see variables in the debugger?

    Some .class files are not compiled with debugging information. For example, many class libraries contain classes that do not have debug information. Without this information, the Java debugger cannot display variables.

    By default, all Java files compiled with the Eclipse JDT include debug information.

  26. How can I change my key bindings?

    You currently can't change your key bindings.

    NOTE: The keyboard shortcuts for the Eclipse SDK can be found here.

  27. How do I uninstall eclipse?

    Simply delete the "eclipse" directory structure where you unzipped eclipse. Eclipse does not modify the registry so simply removing the files will fully uninstall it.

Plug-in Development: top
  1. What build should I download?

    If you are building tool plugins designed to run on a previously released version of the Eclipse Platform, you should download the corresponding release of the Eclipse SDK. However in most cases plug-in developers will want to use the most recent stable build of the Eclipse SDK. These builds are used by the eclipse developers, and will have the most recent bug fixes and features. By using the latest stable build tool builders can stay on top of the latest changes and improvements and provide feedback, bug reports, suggested fixes, and contribute to the evolution of eclipse. You can find the latest builds on the downloads page. A notice will be posted to the newsgroup when a new stable build is available.

  2. How does the Eclipse SDK build process work?

    Nightly builds of the Eclipse SDK are performed by building the current contents of the HEAD stream of the projects in the Eclipse Project CVS repository.

    Integration builds are performed on Tuesdays at noon EST. They are built using versions of projects located in the CVS repository. Each component team submits which version should be included in the build. The goal of the integration build is to create a stable build. Discussion about the problems with, and patches for, integration builds can be found on the eclipse-dev mailing list.

    The eclipse developers use each integration build to continue development. If the developers feel comfortable using an integration build, it is promoted to stable build status and it is posted to the builds page as a stable build.

    Release builds represent the culmination of a release cycle. They are rigorously tested stable builds that are designed for production use.

  3. How do I use the Eclipse SDK to write my plug-in?

    The self-hosting documentation outlines how you can use the Eclispe SDK to write your plug-in.

  4. Where is the Eclipse SDK documentation?

    There are integrated documentation manuals in Help->Help Contents. You can select a manual from the combo box. These documents are also available online.

    The Eclipse Platform technical overview describes the eclipse architecture and gives a case study on using the architecture.

    There are many articles written by eclipse developers and other contributers that drill down into specific technical problems related to building tool plug-ins for eclipse. These articles are generally very practical in nature and many come with example code.

    The javadoc is available as part of the integrated documentation the latest version is also available online. To view the javadoc for the particular eclipse build you have installed:

    1. Help->Help Contents.
    2. Select the Platform Plug-in Developers Guide from the combo box.
    3. Select Reference->API Reference in the help manual tree.

    Finally, check the component pages for the latest design documents and other notes. The component pages can be found within each of the subprojects of the Eclipse Project page.

  5. Where can I find plug-in examples?

    Example plug-ins are provided as a separate download with each build. If you select a specific build from the downloads page, you will see a section labeled "Example Plug-ins".

  6. I'm having trouble implementing my plug-in. Who should I ask for help?

    If you are encountering problems (related to the eclipse code base) while implementing your plug-in, you can often get help from other members of the eclipse community.

    Start by checking old posts to the newsgroup. You can either use your news reader to search the messages or you can search the news archives (set the search scope to "News Archives"). If you can't find an answer to your question in old newsgroup posts, you can search the mailing lists and the documentation.

    If you still can't find an answer to your question, post your question to the newsgroup. But before participating in the newsgroup, you should read "How to ask questions the smart way" by Eric Raymond.

  7. How do I find out what is currently being developed in a component?

    Check the component web page. It will detail the work that is currently being done in the component and future plans the team has for the component. Component web pages can be found under the Subproject web pages of the Eclipse Project page.

    You can also read the mailing list archives. (If you don't already have one you will need a news/mail archive password).

  8. Where can I find out what API has changed?

    Check the javadoc. Changes to the API will be annotated with a @since tag.

  9. I have a Swing application. Can I integrate it into eclipse?

    Yes you can - but there are some significant limitations, and the integration is not as tight as when you build tools using SWT. The best way to integrate tools into eclipse is to use SWT. To understand this, let's look at the kinds of tool integration that are possible with the Eclipse Platform:

    In-Place-Java-Tools coded using the Eclipse Platform API's using JFace/SWT. This is the best integration approach and the mechanism that most eclipse tools use. These tools run in the same VM and same class libraries as the platform itself and can leverage the full integration API for seamless integration. These tools open in-place in the eclpse workbench and contribute views or editors into the workbench perspectives. Note that an In-Place-Java-Tool is coded in Java, but the Java code can also make use of an ActiveX control as part of its implementation (on the Windows platform only).

    Launch-Java-Tools coded in Java, using Swing or AWT.  These tools run in the same VM and same class libraries as the eclipse workbench itself, but are otherwise loosely coupled and operate in much the same manner as External-Launch-Tools. This mechanism allows swing-in-process tools to be opened as separate windows (not in place).

    External-Launch-Tools coded in any language, that are launched as external programs. These tools open as separate windows and are not visually integrated into the workbench. The platform provides automatic launching of external editors based on file type associations provided by the user or obtained from the underlying operating system.

    In-Place-OLE-Documents (Windows platform only) allow standard OLE enabled tools that support in-place OLE documents (such as MSWord, MS Powerpoint, etc) to be invoked in place in the eclipse workbench to edit documents of the corresponding type. These tools are otherwise loosely integrated, and functionally they are the same as External-Launch tools i.e. they open on a document, edit it, save it.

  10. What about swing-in-place?
    You can have a look at the SWT_AWT class which is an experimental mechanism that can be used to integrate swing widgets into eclipse in-place. However, there are some significant limitations associated with this:
    • It only works on Windows.  Swing/AWT ships with a library that is linked in such a way that it does not export its X/motif libraries and this causes a problem where globals are concerned since SWT tries to use the same libraries.
    • It does not work with light weight widgets.  There are issues with the keystrokes getting sent to the light weight widget.
    • It is experimental (in the internal package org.eclipse.swt.internal.awt.win32) and therefore the API may change.

  11. What if I have existing swing code?

    SWT was designed to be small, fast, and leverage the native widget system available on each platform. In the area of high performance workstation tools, we believe this is very important. If you want native look and feel and good platform integration, then SWT is a good choice... and SWT is the best approach for integrating tools into the eclipse platform. For the best results Plug-in developers targeting tight eclipse integration should use SWT.  Plug-in developers with existing swing based components should in most cases use the Launch-Java-Tools approach described above (yielding swing-in-process) as a stepping stone towards ultimately achieving tighter tool integration. The experimental internal mechanism for swing-in-place should be used only after considering the risks and limitations as indicated above.

  12. How do I use eclipse to develop eclipse?

    The self-hosting instructions explain how to use eclipse to develop eclipse.

    • If you want to work with the current version of the eclipse code, you will need to connect to the Eclipse Project CVS repository. To connect to the Eclipse Project CVS repository, open the CVS repositories view (Perspective->Show View->Other...->CVS->CVS Repositories) and create a new CVS repository location (right click->New->CVS Repository Location. Enter the following information into the "Add CVS Repository" dialog.
        Connection type: pserver
        User name: anonymous
        Password: <leave empty>
        Host name: dev.eclipse.org
        Repository path: /home/eclipse

    NOTE: When you are connected as anonymous you will have read rights to the repository, but you will not be able to commit any code.

  13. How do I submit a fix for a bug?

    While using the Eclipse SDK to develop your plug-in, you found a bug in eclipse. You submitted a bug report, but need the fix now. You've debugged the problem, and there is a simple fix. So you figured out how to use eclipse to develop eclipse and have written a fix for the bug you found. Now you want to release the fix to the eclipse community. How do you do this?

    First, create a patch. You can create a patch by using the Team patch creation facility.

    1. Select the project you have patched. It must be connected to the Eclipse Project CVS repository.
    2. Right click->Team->Create Patch...
    3. The Create Patch wizard will prompt you for a file name. You should name your patch with the bugzilla bug report id.

    Now you can submit the patch to the appropriate component developer mailing list. Be sure to include information about what your patch fixes. The committers for the component will evaluate your patch to see if it fixes the bug and is acceptable. If your patch is accepted, it will be released by the component team into the repository.

  14. In addition to submitting fixes, how else can I contribute to the Eclipse Project?

    One initially thinks of submitting patches and enhancing components as the best way to contribute. However, there are many other valuable ways to get involved and help.

    • You can participate in the newsgroups. There are many questions that arise on the newsgroup; you can use your eclipse experience to help others.
    • You can report bugs.
    • You can develop your own plug-ins and provide feedback based on your experience.
    • You can write articles.
    • But the most important contribution you can make to the success of eclipse is to build real, useful tool plugins for eclipse that solve real problems for users - either as commercial products or open source projects.

Miscellaneous: top
  1. Where can I get more plug-ins?

    The eclipse.org site includes a project called The Eclipse Tools Project. The mission of the Eclipse Tools Project is to foster the creation of a wide variety of tools for the Eclipse Platform. Although some experimental plug-ins can be found on component pages, eclipse.org does not provide a generic area for the community to share plug-ins. But there are already some plug-in projects on http://www.sourceforge.org/, and in the future eclipse.org will provide a way to locate other tool plugins designed to work with eclipse.

  2. What other programming languages does the Eclipse Platform support?

    The Eclipse Platform is a universal tool platform. It has no knowledge of any specific language, but opens the door for plug-ins to support various languages. Similarly, the platform provides support for building source editors for specific programming languages but it doesn't actually include those editors.

    The Eclipse SDK includes a full featured Java IDE. This is an example of how to extend the platform to support a particular language. The SDK provides editors for Java files and for plug-in XML files.

    If you are interested in tools for other languages, you should watch (or join) The Eclipse Tools Project, which already has work on a C/C++ Development Tool plugin (CDT) underway.

  3. Is there a Tomcat plug-in?

    No, the Eclipse SDK does not include a Tomcat plug-in.
