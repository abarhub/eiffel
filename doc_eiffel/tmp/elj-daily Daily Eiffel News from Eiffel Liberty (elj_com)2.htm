ELKS 2001 STRING class is now an official standard
( elj-ID 797 ) ( Created Sep 02, 2001, 22:04 UTC ) ( By Roger Browne ) ( Seen at comp.lang.eiffel ) ( Submit by Geoff )

The Eiffel Library Kernel Standard STRING class, 2001 Vintage, has now been adopted as an official NICE Standard, identified as "NICE-4".

This standard has been unanimously approved by polls in each of the following groups:

  • The libraries group that worked to produce it
  • The NICE Board
  • The members of NICE

This standard, together with some explanatory notes, is appended to this message.

All NICE library standards can be downloaded from:

http://www.eiffel-nice.org/standards/

Thanks to all those who worked hard over the past two years to make this latest NICE standard possible. We look forward to its implementation in the coming releases of Eiffel compilers, and to the day when a piece of Eiffel code can run under any Eiffel compiler without change.

Regards,
Roger Browne
(NICE Secretary)

[SmallEiffel] Beta-2 version of thread-safe SE (over SE-0.75)
( elj-ID 798 ) ( Created Sep 02, 2001, 22:35 UTC ) ( By Miguel Oliveira e Silva ) ( Seen at comp.lang.eiffel ) ( Submit by Geoff )

The thread-safe patch over se-0.75, passed all my tests so here it goes.

There are now two possibilities to enable multithreading (using POSIX threads in UNIX [tested only in linux]): using the same old command line option (-pthread), or using the default ace option (as happens with ISE's ace files):

multithreaded (yes);

All examples have now an appropriate ace file.

The new release can be found at:

here and here

It can also be found (together with [almost] all past releases and SE's versions) at:

ftp://ftp.ieeta.pt/~mos

[ePalm/SmallEiffel] ePalm upgraded to use SmallEiffel -0.75
( elj-ID 799 ) ( Created Sep 02, 2001, 22:41 UTC ) ( By Victor B. Putz ) ( Seen at comp.lang.eiffel ) ( Submit by Geoff )

The CVS tree of ePalm has been "upgraded" to use SmallEiffel -0.75 as its base. This has not been fully tested--in particular, use of the GC has only been ad hoc tested...

But it looks great! Memory is released correctly to the OS, and executable sizes are on average about 10k smaller (which for the PalmOS is still a good deal). Minor changes to the runtime... (and I'm not sure my bastardized epalm_calloc function will do what it needs to). The "HASH_TABLE_SIZE" method for prime number cielings was modified to only return values below 32767; ePalm uses 16-bit INTEGERs. Hopefully this won't create too much of a problem.

I could still wish that I could massage the size of the output C files, but overall I'm very pleased (probably what will happen is that I'll write another Ruby build script to chop a single output file into multiple source files; perhaps something along the lines of an epalm_compile.rb script which will handle all the makefile confusions).

In any case, changes are only made to the CVS tree. I'll go for another .tar.gz file (which somehow turned into just a plain ol' .tar file!) after more testing.

SE -0.75 looks great. Can't wait for SE's implementation of agents... (er--that IS coming, isn't it? Hope it doesn't balloon executable size!).

[ISE Eiffel] O-O database: Eiffel-MATISSE Binding
( elj-ID 800 ) ( Created Sep 02, 2001, 22:44 UTC ) ( By Eirik Mangseth ) ( Seen at comp.lang.eiffel ) ( Submit by Geoff )

I've received the following from ISE and take the liberty to post it here:

"Developers in search of a great object-oriented database solution that works with Eiffel will be happy to learn about the upcoming release of a new Eiffel to MATISSE binding for EiffelStudio 5.0. MATISSE 5.0 is, according to its designers, the first database to integrate object technology and standard SQL. It is designed for object developers and database programmers who want to boost application performance and reduce development time. Based on technology used to run mission-critical applications for more than a decade, MATISSE 5.0 is architected for performance, scalability, and flexibility, and fits nicely with the Eiffel solutions for high-quality O-O development."

I not 100% certain, but I believe this release is a combination of ISE's previous binding merged with Thomas Beale's work on a Matisse binding.

An Interoperable Knowledge Methodology for Future-proof Information Systems
( elj-ID 803 ) ( Created Sep 02, 2001, 22:54 UTC ) ( By Thomas Beale ) ( Seen at http://www.deepthought.com.au/ ) ( Submit by Geoff )

A two-level modelling approach to system building, based on "archetypes" is proposed as for information system development, particularly in domains like health, where domain knowledge is complex and voluminous. This methodology is being developed not just on paper, but in the GEHR project, with a real implementation.

The paper - An Interoperable Knowledge Methodology for Future-proof Information Systems describes this approach in detail, and is available in HTML and PDF (1.2 Mb)
